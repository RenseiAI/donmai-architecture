## Context

The OSS execution layer ships seven fused agent-runtime providers (`claude`, `codex`, `gemini`, `agy-cli`, `ollama`, `opencode`, `amp`). Each one **fuses three independent facts into a single opaque id**: *which agent loop drives the turn* (the harness), *which company's model answers* (the endpoint), and *over which wire protocol / serving host / auth mode* it runs (the transport). That fusion has three concrete costs:

1. **Adding a capability is a renegotiation, not a one-line add.** "The Google plugin gains Antigravity-CLI support" today means editing a fused provider, the `applyRuntimeProviderRewrite` special-case in the platform resolver, and several hand-maintained registry maps in lock-step. It should be one new table row.
2. **Cost and transport are per-provider defaults, not config decisions.** `claude` defaults to a host-subscription CLI; `applyRuntimeProviderRewrite` imperatively rewrites `gemini + (local|host-session) → agy-cli` *after* the access gate. The provider, not the org/project/node/machine config, decides what a run costs.
3. **The north star is inexpressible.** `harness=OpenCode × model=Google-served-by-Ollama × sandbox=E2B × tracker=Linear × vcs=GitHub` cannot be built, because OpenCode is welded to a provider id rather than to the wire protocol it actually drives.

The five owner requirements driving the redesign: (#1) a third party contributes a model or a harness via a signed manifest; (#2) models are named **by company** (Anthropic / OpenAI / Google), never by harness — `gemini-cli` stays deleted and a user choosing Google-via-Antigravity sees "Google"; (#3) config selects `(harness × endpoint × host × auth × model)` per org/project/node **and per machine + per workload**; (#4) cost/transport are resolved from scheduler config, not a provider default; (#5) one hosted capability matrix covers LLM providers **and every peer family** from a single source of truth.

The v1 design (`01-current-state-and-plan.md`) assumed a single fused provider with N transports. Its *primitives* — the signed pre-load manifest, `OneShotProvider.Complete`, `SpawnComplete`, `provider-matrix.json` codegen + CI parity, and the `OnPreSpawn` P0 credential hop — are correct and carry forward verbatim. v2 re-homes them onto a two-axis model. This ADR is `shared` because the decomposition **adds a 9th plugin family** to the OSS base contract (`002`) and **re-homes config resolution + per-machine narrowing** onto the platform's model-access matrix (`rensei-architecture/ADR-2026-06-02`), with a narrow-only enforcement read-path ported into OSS Go.

## Decision

Decompose the LLM layer into **two orthogonal plugin families** plus the unchanged peer families, bind them per-run, generate the capability matrix from one SoT, resolve cost/transport from config, narrow per-machine fail-closed in OSS Go, route the one-shot lane through the harness axis, and let third parties contribute a signed manifest row. Seven sub-decisions:

### D1 — Two axes: a renamed HARNESS family and a NEW 9th MODEL-ENDPOINT family

- **HARNESS** (the renamed `AgentRuntime` family — the family-discriminant string stays `agent-runtime`, byte-identical to `002`; only the human surface is renamed) is the agent-loop driver: **Claude Code, Codex, OpenCode, Antigravity, Amp, `raw`, `stub`**. Its manifest declares agent-loop capabilities (`inject`/`resume`/`tools`/`mcp`/`subagents`/`oneShot`/`nativeJsonMode`) **and a DRIVE surface**: `Drives []WireProtocol` (which wire protocols it can speak to a model) and `DrivesHosts []ServingHost` (which serving styles it can reach). A harness is no longer "Claude"; it is "a driver that can drive these protocols via these serving styles." `Manifest()` replaces the instance-only `Capabilities()`; `Spec` carries a resolved `EndpointBinding` instead of a fused `Model` string.
- **MODEL-ENDPOINT** (NEW, named **by company**) is **Anthropic, OpenAI, Google, Local**. Its manifest declares `Speaks []WireProtocol`, a list of `HostDesc` cells (each a `(serving-host × protocol × authModes × bringsOwnAuth × needsApiKey × costModel × envKeys)` tuple — this is where Bedrock/Vertex/direct/OAuth-CLI differ for the *same* company, and where one company carries two protocols, e.g. Google's native `gemini-generate` plus `openai-chat` via the Ollama `/v1` shim), and a `Models []ModelDesc` catalog. It never runs an agent loop. Its sole verb is `Resolve(EndpointRequest) → EndpointBinding`, turning a scheduler-chosen `(model, host, auth)` — all already resolved from config — into a concrete binding (`company, model, baseURL, protocol, host, auth, costModel, bringsOwnAuth, env`). It is consumed by a harness, or called directly by the one-shot / direct-api lane.

The AuthMode vocabulary is the canonical **5-mode** set (`byok | metered | shared | host-session | local`), OSS-canonical so the scheduler and the manifest speak one vocabulary. There is no `api-key` mode: USER-supplied keys are `byok`, Rensei-funded keys are `metered`.

### D2 — Binding + validity rule

A **RUN** is the product of **one HARNESS cell × one MODEL-ENDPOINT cell + the peer families** (sandbox, issue-tracker, version-control), with **every choice resolved from config**. A `(harness × endpoint-host)` pair is **valid iff `harness.Drives ∩ {endpoint host's Protocol} ≠ ∅` AND `host ∈ harness.DrivesHosts`** — the harness natively speaks the host's wire protocol, with **no implicit translation**. **Only same-protocol cells are valid.** A cross-protocol pairing (e.g. OpenCode's `openai-chat` against an Anthropic `anthropic-messages` host) is not emitted, because such a translation has no declared owner, host, or cost; therefore `anthropic-messages` is **not** in OpenCode's `Drives`. Cross-protocol gateways are a **future first-class ModelEndpoint host kind** (a translating-gateway host that owns the shim + its own serving host + cost attribution); until one ships, no cell crosses protocols. Validity is asserted twice (defense-in-depth, mirroring today's double `enforceAccessPolicyGate`): the platform scheduler pre-filters candidates from the matrix (fail-fast UX), and the OSS runner re-asserts `assertValid` at bind time (OSS-standalone correctness).

### D3 — The capability matrix is the set of valid cells, generated from one SoT

`capability-matrix.json` is the **only** hand-authored capability data, generated by `go generate` in OSS donmai from per-provider Go `Manifest` literals (`harnesses.json` ← harness manifests, `endpoints.json` ← endpoint manifests, `matrix.json` ← a hand-authored valid-cell list + denylist). It carries `harnesses[]`, `modelEndpoints[]`, `harnessEndpointCells[]` (one row per **valid** cell, each carrying resolved `protocol`/`host`/`transport`/`authModes`/`costModel`/`oneShot`/`nativeJsonMode`/`structuredVia`/`legacyProviderId`/`stability`/`smoked`), and one section per peer family (`sandbox`, `issueTracker`, `vcs`, `deployment`, `agentRegistry`, `kit`). Everything downstream is generated from it: both Go registries (killing the hand-forked rensei-tui daemon registry — `feedback_rensei_daemon_run_fork_rot`), the platform maps (`PROVIDER_REGISTRY`, `KIND_TO_ENV_VAR`, `detect-provider-type`, `access_policy` seed wildcards), the hosted docs page (one section per family into `donmai-site` for OSS families, `rensei-docs` for platform cells), and the editor palette. A **load-bearing CI parity gate** — a single pure test runnable with `GOWORK=off` — blocks merges on: the JSON being byte-identical to a fresh `go generate`; every `legacyProviderId` existing in both Go registries and `PROVIDER_REGISTRY`; `cell.protocol ∈ (harness.Drives ∩ {host protocol})`; `cell.authModes ⊆ host.authModes` and each declared in some `access_policy` baseline; `AGENT_ENV_BLOCKLIST` de-duplicated; and per-cell `caps` narrowings only *removing* capabilities (`cell.caps ⊆ harness.caps`). Adding `(antigravity × google)` is one new row in `matrix.json` — no edit to either axis list or any platform map. A `contractAbi` skew-tolerance window keeps an OSS-side cell add from reddening an unrelated platform PR; full lock-step applies only when the ABI moves.

### D4 — Config-resolved cost/transport (plugins declare; the scheduler selects)

Plugins **declare** cells (auth modes, cost model, host, protocol); the scheduler **selects** one from the existing two-axis cascade — **no new resolver invented**. The assignment cascade (the platform model resolver: node > project > org > system-worktype > system-default, most-specific tier wins outright, `basedOn` fills unset fields) yields the harness + endpoint (company) + model + candidate authMode set. The access-matrix cascade (`resolveEffectiveAllowedModes`, narrow-only intersection of org ∩ project `access_policy` per `(company|model, authMode)` cell) narrows it. `selectAuthMode` picks one concrete mode (fail-closed). **Transport is then derived from the chosen `(harness × host)` pair, never a provider default; cost is the chosen cell's `costModel`.** `applyRuntimeProviderRewrite` is **deleted and becomes manifest data**. Output **strictness is config-resolved per workload, not a global guarantee**: subscription / `BringsOwnAuth` cells deliver **SOFT JSON** (drive-to-terminal, append the schema instruction, validate-repair-then-drop on `!SchemaOK` — the accepted KG posture of ADR-2026-06-03); keyed (`byok`/`metered`) cells deliver **STRICT native** structured output (`json_schema` / `responseSchema` / `response_format`, server-constrained). The org/project/workflow-node config chooses the cell and therefore the strictness↔cost tradeoff. Cost is **dual-stamped**: `cost_events` records the **endpoint (company+host) as the PRIMARY attribution key** (it is what bills) **and the harness as a sibling column**. Cost honesty is structural: a `BringsOwnAuth` cell skips key injection entirely, so it cannot simultaneously receive a metered key — a provider author cannot lie about cost to get cheaper routing.

### D5 — Per-machine narrowing: HYBRID, daemon-side, narrow-only, fail-closed

Because no current dispatch shape binds the target host at resolve time (pool/unrouted *and* directed dispatch both bind the host only at claim — verified against the platform session-creation and execution-capacity paths), the platform cannot bake the machine-narrowed cell before dispatch. The machine tier is therefore enforced **daemon-side**, by a pure OSS Go package `donmai/runner/access` exposing `ResolveMachineCell`, ported from the platform narrow-only core (`mergeMatrices('narrow-only')`, `resolveAccessCell`, `resolveEffectiveAllowedModes` — ~200 lines, zero DB, zero SDK). The gate runs **one step before `OnPreSpawn`** (the credential hop): `claim → RunSpec → [NARROW-ONLY GATE] → OnPreSpawn(InjectSnapshotIntoEnv)`. On deny the daemon NACKs the claim and **never reaches key injection** — a denied keyed cell can never receive a key. The enforcement is **intersection-only and FAIL-CLOSED**:

`effective(machine, workload) = platformAllowed(org ∩ project)  ∩  machineAllowed(machine, workload)`

where `platformAllowed` is the **immutable ceiling** — a *closed, already-evaluated set*, not a raw matrix, so the `mergeMatrices` inherit-all branch cannot be exploited to broaden. A machine block can only **subtract**; the worst a buggy/stale/malicious daemon block can do is shrink the set to ∅ (over-restrict, degrading availability), **never grant a cell the org/project policy did not already allow**. A `nil` machine block ⇒ `effective = platformAllowed` (safe). The new policy rides the existing `daemonMutations` → heartbeat `pendingMutations[]` → `applyPendingMutations` sync channel as two new ops (`modelAccess.set` / `modelAccess.clear`) writing a new `ModelAccessConfig` block into `daemon.yaml` (policy/routing only — **never credentials**; keys stay on the snapshot socket). Drift between the TS writer and the Go reader is pinned by a single golden test-vector file (`narrow-only-vectors.json`) loaded by **both** the TS `mergeMatrices` tests and the Go `ResolveMachineCell` tests under one CI parity gate. This is a `shared`-boundary amendment to the platform-only ADR-2026-06-02: authoring/storage/cascade stay platform-only; only the narrow-only **enforcement read-path** moves into OSS.

The fail-closed invariant is the load-bearing security property and is byte-identical across both corpora:

**Narrow-only fail-closed invariant.** For any machine `M`, workload `W`, and cell `(company, model, authMode)`:
`effective(M, W) = platformAllowed(org, project) ∩ machineAllowed(M, W)`, where `∩` is set intersection and `platformAllowed` is the immutable ceiling. A machine block can produce a result `⊆ platformAllowed` and never `⊋ platformAllowed`. The worst a buggy, stale, or malicious daemon block can do is shrink the set to ∅ (over-restrict, degrading availability); it can never grant a cell the org/project policy did not already allow. A `nil` machine block ⇒ `effective = platformAllowed`. The ceiling is passed as a closed, already-evaluated set — not a raw matrix — so the `mergeMatrices` inherit-all branch cannot be exploited to broaden. The same golden `narrow-only-vectors.json` is honored by both the TypeScript writer and the OSS Go enforcement reader under one CI parity gate.

### D6 — One-shot lane on the harness axis

`OneShotProvider.Complete(OneShotRequest) → OneShotResult` is the non-interactive, schema-constrained completion primitive that **both arch-intel drift and KG extraction** consume. The request carries a resolved `EndpointBinding` (same binding currency). A harness gets it **free** via `SpawnComplete` (drive an interactive session to terminal, buffer complete `AssistantTextEvent`s — never per-token deltas, preserving the anti-spam rule — and `extractAndValidate` against the schema); native-JSON harnesses (`raw` over Gemini/Ollama `/v1` with `NativeJSONMode:true`) override with a direct `Complete()`. **No vendor-SDK import on the cheap path**: arch-intel stops hard-coding `anthropic-sdk-go` as an 11th direct-LLM bypass, and KG's batch handler stops branching on provider — it requests structured completion and the cell resolves the strategy (soft under subscription, strict under a keyed cell). The KG host-session-first ladder becomes an ordering over `HostDesc.AuthModes` (BringsOwnAuth cells first, NeedsAPIKey last), not a second hand-maintained array.

### D7 — Third-party contribution via a signed manifest row

A signed `015` plugin manifest contributes `harness` and/or `model-endpoint` rows plus the valid cells against existing axes; the registry computes the `(harness × endpoint)` cells from the protocol intersection — never hand-authored. **The safest path requires zero Go and never hands the vendor a key**: `entry.kind: remote` + `protocol: openai-compat` is driven by the OSS OpenAI-compat client, and for a `remote` endpoint Rensei holds the key and calls *out* to the vendor's `baseUrl`. Trust bends by entry kind (`static` = Rensei-authored only; `npm`/`binary` = `signed-by-allowlist` + conformance-probed, `binary` sandbox-mandatory for non-allowlisted signers; `remote` = `signed-by-anyone` acceptable). A **two-tier registry** keeps the product matrix trustworthy: `verified` (Rensei-probed, allowlisted, in the curated matrix) vs `community` (signed, installable, flagged unverified, excluded from the curated matrix). "The Google plugin gains Antigravity-CLI" is exactly one `harness` array entry declaring `drivesProtocols:[gemini-generate-content]`; the `(Antigravity × Google)` cell appears automatically — no renegotiation, no Rensei code change, no lock-step release.

## Consequences

### Positive

- **The north star is expressible by construction.** `harness=OpenCode × model=Google-served-by-Ollama × E2B × Linear × GitHub` resolves: Google's `local`/Ollama `/v1` host cell is `openai-chat`; `openai-chat ∈ opencode.Drives ∧ local ∈ opencode.DrivesHosts`; the cell binds. Nothing in OpenCode references "Gemini" — it references the protocol it drives.
- **Adding a capability is one row.** `(antigravity × google)` is a single `matrix.json` entry; the parity gate validates it, the JSON gains the row, docs and editor pick it up automatically. No edit to either axis list or any platform map.
- **Cost and transport become data.** `applyRuntimeProviderRewrite` — the only imperative provider-rewrite in the resolver — is deleted. "Subscription-cheap-first" becomes the org/project matrix preferring host-subscription cells (or a workload pinning the opposite, as KG does). Cost honesty is structural, not policy.
- **Naming-by-company is satisfied.** `gemini-cli` stays deleted; `gemini` and `agy-cli` collapse to **one company (Google)** with two host cells; the user sees "Google" + a harness, never "antigravity" as a model name.
- **One hosted matrix for every family.** "Issue-trackers aren't created equally" becomes a legible per-flag column set (Linear's all-true row vs Jira/Asana/Monday sparse rows). Docs, editor palette, and runtime resolver are the *same generated cells*, so an operator can predict exactly which cell a node+policy resolves to.
- **The forked rensei-tui daemon registry becomes generated**, removing a standing rot source.
- **Per-machine flips need no redeploy.** The Mac Studio runs host-session SDLC and flips, for the KG workload only, to byok → Haiku on Vertex — pushed as one synced config mutation, applied on the next heartbeat.

### Negative

- **A thin 9th family.** `ModelEndpoint`'s only verb is `Resolve`, and it is not user-facing — it does not fit `002`'s "families have user-facing verbs" assumption (provision/acquire/clone). We accept the thinness in exchange for uniform hosted-matrix treatment and a clean third-party surface.
- **A codegen tax.** `capability-matrix.json` and the registries become generated artifacts behind a load-bearing CI parity gate that spans three repos (donmai generates, rensei-tui consumes, platform consumes). An OSS cell-add reds the platform parity test until platform regenerates (mitigated by the `contractAbi` skew window).
- **An OSS mirror of narrow-only.** The platform's narrow-only logic now has a second honorer (the Go port). Two implementations of one security-critical rule is inherent logic-drift risk, paid down by the single golden `narrow-only-vectors.json` honored by both sides under one CI gate.

### Risks

- **Narrow-only drift granting a forbidden cell.** Mitigated structurally: intersection-only, the ceiling is a closed set (never a raw matrix), three independent delete-only guards in the TS original are replicated in the Go port, and the golden-vector parity gate fails CI on both sides on any divergence. Worst case is over-restriction (availability), never a confidentiality/cost escalation.
- **Cross-protocol over-advertising.** A future contributor declaring a cross-protocol cell without a translating-gateway host would advertise an impossible pairing; the parity gate (`cell.protocol ∈ intersection`) rejects it at publish.
- **`legacyProviderId` lifetime.** The OSS `ProviderName` string is a wire value in `QueuedWork`; aliases must survive until every `cost_events.provider_id`, default-SDLC reference, and `PROVIDER_REGISTRY` id migrates. Hard-delete is deferred to Phase 7 *after* the default-SDLC migration (the one deletion constraint).
- **The OSS daemon leaves `OnPreSpawn` nil today** (`worker_spawner.go`); per-machine keyed flips are not actuatable until the P0 credential hop lands. Everything in D5 rides on Phase 0.

## Alternatives considered

- **v1 fused-provider (one provider, N transports).** Rejected: it is the status quo whose three costs (renegotiation-per-capability, cost-as-default, inexpressible north star) prompted this ADR. It cannot express `OpenCode × Google-served-by-Ollama` because OpenCode is welded to a provider id, not to the `openai-chat` protocol it drives. Its primitives are kept; its fusion is not.
- **Company-plugin-owns-harness (Option A — bake the harness onto the company plugin).** Rejected: it re-fuses the two axes the moment a company ships more than one harness. Google would have to bundle both Antigravity and the `raw` direct loop into one plugin, and "Google gains Antigravity-CLI" would again be a plugin renegotiation rather than a one-cell add. It also makes `OpenCode × Google` impossible, since OpenCode is owned by no company.
- **Rename-only (Option C — keep one fused family, just rename `AgentRuntime` → `Harness`).** Rejected: a rename does not separate *which loop drives* from *which company answers*. Cost/transport stay per-provider defaults, `applyRuntimeProviderRewrite` survives, and the north star is still inexpressible. The decomposition — not the label — is what makes "OpenCode drives Ollama-served-Google" expressible without any provider knowing about another.

Each alternative fails the same test: the `OpenCode × Google × Ollama` north star is expressible only when the harness binds to a *protocol* and the company is a *separate, host-bearing axis*.
